知识点–概念：Oracle Database Release Numbers 新版本号规则

1. 介绍
    # 从 201707 开始
    # 从 18c 开始  
    # 之前为多年一发布
    # 之后为每年发布一个年度发行版 Release，之后追加季度更新  
    # 优点  
        - 能够将更多的新功能更快的提供给用户  
        - 大大减少了单次软件变更的数量  
        - 避免大的版本升级对系统产生的风险  
    # 网址
        - https://docs.oracle.com/en/database/oracle/oracle-database/18/upgrd/oracle-database-release-numbers.html#GUID-1E2F3945-C0EE-4EB2-A933-8D1862D8ECE2
2. 格式
    # <Major_Database_Release_version>.<Release_Update_version>.<Release_Updates_Revision_version>.<increment_version>.<reserved>
        - 对于大多数用户来说，只需要关注前 3 位  
    # <Major_Database_Release_version>
        - 表示发行年度  
        - 每年一发布  
    # <Release_Update_version>
        - 季度更新 RU 版本
        - 每年 1/4/7/10 月发布  
        - 主要包含查询优化器相关 BUG 和安全漏洞的修复等  
    # <Release_Updates_Revision_version>
        - 季度更新修订 RUR 版本
        - 对每一个 RU，概率性提供更新修订 RUR  
        - 对 RU 的缺陷进行的修复，并包含最新的安全漏洞补丁  
    # <increment_version>
        - Oracle 数据库的增量版本，偶尔会用于 Oracle 云数据库中  
    # <reserved>
        - 保留位，预留给将来  
3. 维护期  
    # 每个 Release 发布后的至少两年内，都会提供 RU 的支持  
    # 每个 RU 会提供两个独立的 RUR 支持  
    # 每个 RUR 之间间隔 6 个月  
    # 因此每个年度 Release 至少会有 3 年的支持周期

知识点–概念：备份恢复基本概念

    1. 高峰时期用容灾系统做备份，非高峰期用 RMAN 做备份  
    2. 关键性文件和非关键性文件  
        # 关键性文件，丢失后会导致数据库宕机  
            - 系统表空间，UNDO 表空间  
        # 非关键性文件，丢失后不会造成数据库宕机  
            - 用户数据表空间  
            - 非关键性丢失的修复可以直接在线完成
    3. 数据库可恢复性  
        # 控制文件多路复用  
            - OS 文件存储  
                配置 control_files 参数，添加控制文件数量  
                控制文件分别放在不同的物理存储上  
            - ASM 存储  
                每个 disk_group 上都放一个控制文件副本  
        # 日志文件多路复用  
            - 创建多个日志组
            - 每个日志组至少有两个成员  
        # 数据库为归档模式  
        # 数据库有备份

知识点–概念：备份方式

    1.整体备份和部分备份  
        # 整体备份 whole backup
            - 备份整个数据库
            - 包含全部数据文件和至少一个控制文件  
        # 部分备份 partial
            - 备份部分数据文件
            - 可能包含控制文件  
    2.冷备份和热备份  
        # 冷备份 cold backup
            - 数据库一致性关闭后进行的备份
                RMAN 做冷备份需要在 MOUNT
                手动冷备份可以在 CLOSE
            - 冷备份是一致的  
            - 冷备不需要归档  
            - 冷备份只适用于小型不重要的业务，不适用于 OLTP 类型的对连续运行要求高的场景  
        # 热备份 hot backup  
            - 数据库在 OPEN 状态进行的备份
                手动无法热备份
            - 热备份需要归档模式  
            - 备份文件加归档日志加 redo 日志，可以恢复到故障时刻，不会数据丢失

知识点–概念：完全恢复和不完全恢复

    1. 完全恢复 complete recovery  
        # 数据库可以恢复到发生故障的时间点  
        # 已提交的数据不会丢失  
        # 备份文件，归档日志，日志文件  
        # 不用 resetlog  
    2. 不完全恢复 incomplete recovery  
        # 数据库只能恢复到发生故障的时间点之前的某个时刻，从恢复时间点到故障时间点的操作全部丢失  
        # 可以指定恢复到的时间点、还原点、SCN、日志序列号  
        # 通常因为某个归档日志错误，或在线日志错误  
        # 需要 resetlog  
    3. 三种不完全恢复  
        # 在 12C 版本的情况下，三种不完全恢复在 PDB 中已经不适用了，因为 12C 强调的是 PDB 的恢复，CDB 不过是 PDB 的容器  
        # 基于用户控制的不完全恢复
            - RECOVER DATABASE UNTIL CANCEL ;  
        # 基于 SCN 的不完全恢复
            - RECOVER DATABASE UNTIL CHANGE <scn> ;  
        # 基于时间的不完全恢复
            - RECOVER DATABASE UNTIL TIME '<date_time>' ;

知识点–概念：还原和恢复

    1. 还原 restore  
        # 物理上的恢复，文件级的恢复  
        # 将物理文件 copy back  
        # 恢复后，数据库的时刻点前进到备份时的时刻点  
    2. 恢复 recover  
        # 逻辑上的恢复，数据级的恢复  
        # 利用日志进行重做数据文件，读取归档日志和在线日志，将备份时刻点到故障时刻点之间的数据加载回数据文件  
        # 恢复后，数据库的时刻点前进到故障发生的时刻点

46.1.2 增量备份

知识点–概念：增量备份 incremental backup

    1. 全备份 FULL BACKUP
        # 指定数据文件的全部 block  
        # 若不指定增量备份，则默认进行全备份
        # 全备份不会被用作增量备份的 level 0 备份，不论是 backupset 形式的全备份还是 image copy 形式的全备份
    2. 差异增量备份累计增量备份
        # 差异增量 Differential incremental
            - 默认增量方式
            - 从最近的一次 <=n 级的增量备份以来变化的数据
            - 范例
                level 1 级差异增量备份，备份从上一次 level 1 级增量备份以来的变化
        # 累计增量 cumulative incremental
            - 从最近的一次 <=n-1 级的增量备份备份以来变化的数据  
            - 恢复时的速度更快，读取的备份文件更少
            - 占用的硬盘空间更多
            - 范例
                level 1 级累计增量备份，备份从上一次 level 0 级增量备份以来的变化
    3. 增量备份的级别 level
        # 旧版本，需要先手动创建 level 0，才能创建其他 level 的备份  
        # 新版本，可以直接创建 level n 的备份
            - 如果缺失相应的级别的备份，则此次备份实际创建的是缺失的级别的别分
            - 范例
                缺失 level 0
                第一次备份 level 1，实际备份的是 level 0
                第二次备份 level 1，才是真的备份 level 1
        # 差异增量，level 为 n，如 1、2、3  
        # 累计增量，level 为 nc，如 1c、2c、3c 
        # level 0 级增量备份，可以是 backupset，也可以是 image copy
        # level n 级增量备份，只能是 backupset
            - level n 级增量备份可以基于 image copy 的 level 0 级增量备份
            - level n
    4. 增量备份的字典
        # V$BACKUP_DATAFILE.incremental_level                <-- 记录了备份文件的增量级别

知识点–概念：增量备份的算法

    1. 数据文件 checkpoint scn
        # 每个数据文件都有一个 checkpoint scn
        # 所有低于旧于此 scn 的修改会保证被还原
        # 使用 level 0 增量备份进行还原时，数据文件的 checkpoint scn 变更为 level 0 增量备份的创建时 scn
        # 使用 level n 增量备份记性还原时，数据文件的 checkpoint scn 前进到 level n 增量备份的创建时 scn
    2. 增量开始 scn Incremental start SCN
        # 只应用于 level n 增量备份
        # 通常是 level n 增量备份的父备份的 scn
        # 创建增量备份时，新于此 scn 的 block 会包含到备份中，旧于此 scn 的 block 不会包含到 备份中
    3. Block SCN
        # 每个块的 scn，记录最近一次修改的 scn
    4. 进行增量备份的过程
        # RMAN 确定 Incremental start SCN
        # RMAN 扫描数据文件中的所有的块的 block scn
        # RMAN 将 block scn 高于 Incremental start SCN 的块包含到备份中
    5. 进行具有块追踪的增量备份过程
        # RMAN 确定 Incremental start SCN
        # RMAN 读取块追踪文件，避开没有发生修改的块
        # RMAN 读取剩余的块，检查每一个块的 block 和 Incremental start SCN，将 block scn 高于 Incremental start SCN 的块包含到备份中
    6. NOLOGGING
        # 即使更改是使用 NOLOGGING 选项创建的对象，RMAN 也会在恢复期间应用包含更改数据的所有块
        # 因此，如果还原在进行 NOLOGGING 更改之前进行的备份，则增量备份是恢复这些更改的唯一方法
    7. RMAN 的优化
        # 在进行恢复时，RMAN 优先使用增量备份而不是全备份加归档，因为增量备份的速度更快

知识点–概念：块追踪 Block Change Tracking

    1. 介绍
        # 提高增量备份的性能
            - 未开启块追踪功能时，RMAN 进行增量备份时，需要扫描整个数据文件的所有块
            - 开启块追踪后，RMAN 备份时，通过块追踪文件定位到发生变化的块
        # 只针对高于 0 级别的增量备份生效
            - level 0 增量备份还是需要 RMAN 扫表整个数据文件
        # 默认未开启块追踪功能
    2. 块追踪文件 change tracking file
        # 二进位文件
        # 由 ORACLE 自动生成和维护，不需要手动介入
        # 在生成 REDO 时，将数据文件中发生变化的 block 记录到块追踪文件中
        # 使用 bitmap 记录变化的块
        # 根据最近 8 次备份进行记录
            - 每个备份记录都有一个 bitmap，记录相对于当前时刻的块的变化
            - 新的备份会覆盖掉最旧的一次备份记录的 bitmap
            - 如果增量备份是基于超过 8 次备份以前的备份，则不会用到块追踪
        # 默认由 OMF 管理，存放在 DB_CREATE_FILE_DEST，也可以手动指定位置
        # 整个数据库使用一个块追踪文件
        # 在 RAC 环境中，块追踪文件需要存放在可以被集群内所有节点所访问的共享存储上
        # Oracle 推荐将裸设备用作块追踪文件
        # 块追踪文件不能进行备份和恢复
        # 当数据库发现块追踪文件失效时，会自动重建块追踪文件
        # 进行了数据库恢复或数据库中某个部分的恢复后，数据库会重置块追踪文件并重新开始记录
        # 块追踪文件的大小
            - 追踪文件的大小取决于块的数量和 redo 线程数
            - 单实例环境，估计块追踪文件的大小大约为数据库总量的 1/30000
            - RAC 环境，估计块追踪文件的大小大约为数据库总量的 1/30000 * enabled_thread
            - 初始大小为 10M，自增大小为 10M
            - 每一个数据文件对应最少 320K 的空间
    3. 移动块追踪文件
        # 在 MOUNT 状态下，使用 ALTER DATABASE RENAME FILE 可以移动块追踪文件，并保留块追踪文件的内容
        # 在 MOUNT 或 OPEN 状态下，使用 ENABLE/DISABLE 可以切换块追踪文件，会丢失块追踪文件中的内容
    4. 块追踪的字典
        #  V$BLOCK_CHANGE_TRACKING                <-- 记录是否开启了块追踪功能，以及块追踪文件的信息

知识点–语法：ALTER DATABASE 启动或关闭块追踪功能

    1. 介绍
        # MOUNT 或 OPEN 状态
    2. 语法
        # ALTER DATABASE ENABLE BLOCK CHANGE TRACKING ;
        # ALTER DATABASE ENABLE BLOCK CHANGE TRACKING USING FILE '<file_location>' [reuse] ;
        # ALTER DATABASE DISABLE BLOCK CHANGE TRACKING ;

46.1.5 归档日志

知识点–概念：归档日志

    1.归档日志  
        # 在切换日志时，将 redo 日志进行归档

        # 归档日志会挂起所有 DML 操作  
    2.归档日志的存放位置  
        # 参数 LOG_ARCHIVE_DEST 
        # 参数 LOG_ARCHIVE_DEST_N
            - LOG_ARCHIVE_DEST 和 LOG_ARCHIVE_DEST_N 只能选择一种
        # FRA  
            - 如果 LOG_ARCHIVE_DEST 和 LOG_ARCHIVE_DEST_N 均为空，则默认存放在 FRA 中  
            - <fra_dest>/<db_name>/<instanct_name>/archivelog  
    3.归档日志冗余  
        # LOG_ARCHIVE_DEST 方式  
            - 配置 LOG_ARCHIVE_DEST 参数  
            - 配置 LOG_ARCHIVE_DUPLEX_DEST 参数

            - 配置 DB_RECOVERY_FILE_DEST 参数  
            - 配置多个 LOG_ARCHIVE_DEST_N 参数  
    4.参数  
        # LOG_ARCHIVE_FORMAT                    <-- 指定归档日志的命名，其中必须包含 %s %t %r  
            - <string>        <-- 直接指定的字符串  
            - %s            <-- 日志序号  
            - %S            <-- 日志序号，零填充  
            - %t            <-- 日志线程号  
            - %T            <-- 日志线程号，零填充  
            - %a            <-- activation ID  
            - %d             <-- database ID  
            - %r             <-- incarnation ID  
        # LOG_ARCHIVE_DEST，LOG_ARCHIVE_DUPLEX_DEST  
            - 指定归档日志的存放位置  
            - 只能指定本地地址

            - 和 db_recovery_file_dest 不兼容  
        # LOG_ARCHIVE_DEST_N                    <-- 指定在 DG 环境中的归档日志的存放位置  
            - n 取值为 1-32  
            - n <=10，可以设置为本地，也可以设置为远程  
            - n >10，只能设置为远程  
            - 语法，ALTER SYSTEM SET LOG_ARCHIVE_DEST_1='location=<PATH>' ;  
        # LOG_ARCHIVE_DEST_STATE_N                <-- 对应每一个 LOG_ARCHIVE_DEST_N 参数，指定其状态
            - 对每一个 LOG_ARCHIVE_DEST_N 必须同时指定 LOG_ARCHIVE_DEST_STATE_N  
    5.字典  
        # V$ARCHIVE                            <-- 记录还未归档的 redo 日志  
        # V$ARCHIVE_DEST                    <-- 记录所有的 LOG_ARCHIVE_DEST_N 参数的设置  
        # V$ARCHIVE_DEST_STATUS                <-- 记录所有的 LOG_ARCHIVE_DEST_STATE_N 参数的设置  
        # V$ARCHIVE_GAP  
        # V$ARCHIVE_PROCESSES                <-- 记录 ARCH 进程的信息  
        # V$ARCHIVED_LOG                    <-- 记录已归档的日志，包括已删除的归档日志，信息来自控制文件

48.1.1 故障处理概念

知识点–概念：MTBF、MTTR、RPO、RTO、MAA

    1.MTBF mean time between failures  
        # 平均稳定运行时间  
        # DBA 应该尽量保证 MTBF 提升  
    2.MTTR，mean time to recover  
        # 平均故障修复时间  
        # DBA 应该尽量降低 MTTR  
    3.RPO，RTO，SLA  
        # 根据目标 RPO 和 RTO，确定数据库的备份策略和修复策略  
        # RPO，Recovery Point Objective   
            - 一次故障所能承受的最大数据丢失量  
        # RTO，Recovery Time Objective   
            - 一次故障所能接收的最大停机时间  
        # SLA，service level agreement  
            - 客户服务水平协议  
    4.MAA  
        # MAA Maximum Availability Architecture 最大化高可用体系  
        # EMCC 提供 MAA Advisor  
        # MAA 架构  
            - production site，使用 DRA、FLASHBACK、RMAN 等进行保护  
            - standby size，使用 DG、ADG 实现与 production site 同步  
            - 当 production site 崩溃时，业务立刻切换到 standby site

知识点–概念：数据库的保护级别 data protection solutions

    1.不同 RTO，使用不同的方法，级别越高，价格越贵  
    2.RMAN 和 OSB  
        # 物理数据级别保护  
        # RPO 保护级别为 hours 或 days  
    3.闪回  
        # 逻辑数据级别保护  
        # RPO 保护级别为 minutes 或 hours  
    4.DG，ADG，RAC  
        # 实时级别保护  
        # RPO 保护级别为 secones 或 minutes

知识点–概念：数据库常见故障类型

    1.语句错误 statement failure  
        # 用户输入无效数据，解决方法，表定义约束  
        # 用户进行无权限操作，解决方法，授予用户相应权限  
        # 空间不足，解决方法，调整数据库空间，调整用户 quota，启动断点运行  
        # 应用逻辑错误解决方法，DBA 没有办法，需要与开发者沟通  
    2.用户错误 user failure  
        # 用户误操作，删除行或删除表等  
        # 解决方法，日志挖掘，数据库恢复，闪回恢复  
    3.用户进程错误 user process failure  
        # 用户会话异常断开  
        # 解决方法，PMON 自动释放服务器进程资源，dba 不需要做操作，不过可以监控告警日志，对错误发生的次数进行统计  
    4.网络错误 network failure  
        # 监听错误，  
        # 解决方法  
            - 监听错误，设置监听冗余，重启和配置监听，tnsname 故障转移  
            - 网络硬件故障，设置网络硬件冗余  
    5.实例错误 instance failure  
        # 实例失败，os 故障，服务器掉电  
        # 解决方法，实例恢复  
    6.介质错误 media failure  
        # 硬盘故障，误删文件  
        # 解决方法，通过备份文件进行恢复

    48.6.1 手动备份

知识点–概念：手动备份

    1.操作内容  
        # 一致性关闭数据库  
        # 在 OS 层直接复制相应的文件，包括参数文件、控制文件、数据文件、临时文件、undo 数据文件、redo 日志  
    2.需要进行的查询  
        # v$datafile    ，查询数据文件  
        # v$controlfile    ，查询控制文件  
        # v$tempfile    ，查询临时文件  
        # v$logfile        ，查询 redo 日志  
        # V$ARCHIVED_LOG，查询归档日志  
    3.手动备份的恢复  
        # 如果包含所有的 redo 日志，则直接将所有的参数文件、控制文件、数据文件、临时文件、undo 数据文件、redo 日志复制到相应的位置即可  
        # 如果没有包含全部的  redo 日志  
            - 先将所有的参数文件、控制文件、数据文件、临时文件、undo 数据文件、redo 日志从冷备份中复制到原来位置  
            - STARTUP MOUNT  
            - RECOVER DATABASE UNTIL CANCEL  
            - ALTER DATABASE OPEN RESETLOGS  
    4.backup mode，手动热备份模式  
        # 进入 backup mode 后，oracle 生成的 redo 日志会巨量增加  
            - 在 backup mode 中，oracle 会停止对相应的 datafile 的写  
            - 因为一个 block 是有多个 os 块组成的，如果不停止对 datafile 的写，可能导致 oracle 块的不一致，称为 split block 现象  
            - 在 backup mode 中，对 oracle 块进行写，数据库会将整个 block 存放到 redo 日志中，直到结束 backup mode，因而 redo 日志的数量会急剧增加  
            - redo 日志的切换会非常快，如果造成切换不及时，可能就会导致数据库卡死  
        # 切换手动热备份模式  
            - ALTER TABLEBASE <tablespace_name> BEGIN BACKUP ;  
            - ALTER TABLEBASE <tablespace_name> END BACKUP ;  
            - ALTER DATABASE BEGIN BACKUP ;  
            - ALTER DATABASE END BACKUP ;
