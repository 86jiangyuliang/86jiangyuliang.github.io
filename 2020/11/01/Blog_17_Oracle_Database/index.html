1.1 SELECT 基础语法

1.1.1 SELECT 语句介绍

知识点–概念：SELECT 语句的功能

    1. 选取 selection  
        # 横向选择，选择表中的行  
        # 通过指定条件以筛选检索的行  
    2. 投影 projection  
        # 纵向选择，选择表中的列  
    3. 连接 join  
        # 多表关联  
        # 通过指定表与表之间的联系，将不同表中的数据整合在一起

知识点–语法：SELECT 基本语法

    1. 语法  
        # SELECT <select_list> FROM <table_name> ;  
    2. <select_list>  
        # <column_name>  
        # <function>  
        # <expression>  
        # NULL

知识点–概念：DISTINCT

    1. 介绍  
        # SELECT 语句中只能包含一个 DISTINCT  
        # DISTINCT 只能接受一个 <identifier>  
            - DISTINCT <identifier_1>, <identifier_2>[, ...]，相当于 GROUP BY <identifier_1>, <identifier_2>[, ...]  
        # DISTINCT 可以放在其他函数进行嵌套  
        # DISTINCT 可以接收 NULL，将所有 NULL 合并为一个 NULL  
        # 也写作 UNIQUE  
    2. 格式  
        # DISTINCT <identifier>  
        # DISTINCT (<identifier>)  
        # <identifier>  
            - <column_name>  
            - <function>  
            - <expression>

1.1.2 列别名

知识点–概念：列别名 column alias

    1. 设置列别名的方式  
        # 空格  
        # AS  
    2. 列别名的命名方式与对象的命名方式相同  
    3. 双引号列别名  
        # 引用双引号列别名时，也要加上双引号  
        # 双引号列别名可以区分大小写，可以使用空白字符，可以使用特殊字符，可以使用关键字  
    4. 列别名可以使用的位置  
        # 除了 ORDER BY，其他子句几乎都不可用

1.1.3 SELECT 语句的从句

知识点–概念：SELECT 语句的基本从句，虚拟表

    1. SELECT 的各个从句  
        # SELECT clause  
            FROM clause  
            JOIN clause  
            WHERE clause  
            GROUP BY clause  
            HAVING clause  
            ORDER BY clause  
            OFFSET clause  
            FETCH clause  
    2. SELECT 语句各个从句的执行顺序  
        # FROM  
        # JOIN  
        # WHERE  
        # GROUP BY  
        # HAVING  
        # SELECT 聚合函数  
        # SELECT  
        # DISTINCT  
        # UNION  
        # ORDER BY  
        # OFFSET  
        # FETCH  
    3. 虚拟表  
        # 每个步骤都会产生一个虚拟表  
        # 虚拟表被用作下一个从句的输入  
        # 只有最后一步生成的虚拟表才会返回给调用者，中间过程的虚拟表都对调用者不可用  
        # 虚拟表的错误与检查  
            - 虚拟表允许存在错误，如两列的列名相同  
            - 输入时对虚拟表进行检查，输出时不进行检查

1.1.4 FROM

知识点–语法：FROM 基本语法

    1. 语法  
        # FROM [<schema_name>.]<table_name>[@<db_link>]

知识点–概念：表别名 table alias

    1. 介绍
        # 表别名在 FROM 中定义  
        # 在连接中，只能对基础表定义表别名，不能对连接后的虚拟表定义表别名  
    2. 表别名可以用在 SELECT 语句中的任意从句  
    3. 表别名的用处  
        # 用在 select_list 中，指定需要选取的列  
        # 用在连接中的 ON 连接，指定连接条件  
        # 用在子查询中，尤其是关联子查询

1.1.5 WHERE

知识点–语法：WHERE 基本语法

    1. 语法  
        # WHERE <where_condition>

1.1.6 GROUP BY 和 HAVING

知识点–语法：GROUP BY 基本语法

    1. 介绍  
        # GROUP BY 中不能使用聚合函数  
        # SELECT list 中的项，要么在 SELECT list 中以聚合函数的参数存在，要么写在 GROUP BY list 中作为分组依据  
    2. 语法  
        # GROUP BY {<column_name>|<function>|<expression>}[, ...]  
    3. 特性
        # 括号和先后顺序并不影响分组结果，只影响排序  
            - 以下三者结果相同  
                GROUP BY column_a, column_b, column_c  
                GROUP BY (column_a, column_b), column_c  
                GROUP BY column_a, (column_b, column_c)

知识点–语法：HAVING 基本语法

    1. 语法  
        # HAVING <having_condition>  
    2. HAVING 与 WHERE 的区别  
        # WHERE 是对 GROUP BY 前的虚拟表进行筛选，<where_condition> 中不可以使用聚合函数  
        # HAVING 是对 GROUP BY 之后的虚拟表进行筛选，<having_condition> 中可以使用聚合函数

1.1.7 ORDER BY

知识点–语法：ORDER BY

    1. 语法  
        # ORDER BY {<order_by_item> [ASC|DESC] [NULLS FIRST|NULLS LAST]}[, ...]  
            - ASC                <-- 升序  
            - DESC                <-- 降序  
            - NULLS FIRST        <-- 空值放在最前  
            - NULLS LAST        <-- 空值放在最后  
    2. <ordey_by_item>  
        # 可以是没有出现在 <select_list> 中的项  
            - <column_name>  
            - <column_alias>  
            - <expression>  
            - <select_list_position>

1.1.8 OFFSET 和 FETCH

知识点–语法：OFFSET

    1. 功能  
        # 跳过前面 N 行  
        # 可以不和 FETCH 一起使用  
        # 处理的是 ORDER BY 之后的虚拟表
    2. 语法  
        # OFFSET <N> {ROW | ROWS}  
            - ROW、ROWS            <-- 语法提示词，无功能意义，但不能缺少

知识点–语法：FETCH

    1. 功能  
        # 只返回接下来的前 N 行  
        # 可以不和 OFFSET 一起使用  
        # 常用于 Top-N 查询
        # 处理 ORDER BY 之后再 FETCH 之后的虚拟表
    2. 语法  
        # [<offset_clause>] FETCH {FIRST|NEXT} {<N>| <percent> PERCENT} {ROW|ROWS} {ONLY | WITH TIES}  
            - FIRST、NEXT        <-- 语法提示词  
            - ROW、ROWS            <-- 语法提示词  
            - ONLY                <-- 指定了多少行，就返回多少行  
            - WITH TIES            <-- 必须配合 ORDER BY 子句使用，对排序的最后一个值将相同的行都取出，即使超出了 {<N>| <percent> PERCENT} 的限制

1.1.9 连接 JOIN

知识点–概念：连接

    1. 连接的意义  
        # 将数据分散储存，避免产生冗余数据  
    2. 等值连接、非等值连接  
        # 等值连接 equijoin  
            - 连接条件为等于的连接  
        # 非等值连接 nonequejoin  
            - 连接条件不是等于的连接  
            - 非等值连接虽然连接条件不是等于，但仍要尽量保证对每一行只有一个匹配行，否则容易产生大量冗余数据，一般没有实际意义  
    3. 内连接、外连接  
        # 内连接 inner join  
            - 内连接仅会保留符合条件的行  
        # 外连接 outer join  
            - 外连接会保留符合条件的行和不符合条件的行  
        # 左外连接 left outer join  
            - 保留左表的不符合条件的行  
        # 右外连接 right outer join  
            - 保留右表的不符合条件的行  
        # 全外连接 full outer join  
            - 左表和右表的不符合条件的行都会保留  
    4. 自连接 self join  
        # 左表和右表是同一张表  
        # 自连接必须使用表别名  
    5. 自然连接 natural join  
        # 自然连接是一种特殊的等值连接  
        # 自动寻找两张表中的公共列，自动进行等值连接  
        # 自然连接的前提  
            - table_A 和 table_B 有公共列  
            - 公共列的列名要相同  
            - 公共列的数据类型要相同，或者不同但可以进行隐式转换，否则会报错  
            - 如果有多个公共列，则所有的公共列都会被用作连接条件，并且必须满足每一对公共列的值都相等  
            - 多个公共列时，可能因为某一对公共列的问题而影响结果，或报错  
        # 自然连接的结果  
            - 会将公共列的重复项去重，并将公共列提到表的第一列  
            - 公共列中行的顺序按照左表中公共列的行顺序  
        # 对自然连接的结果进行查询时，如果引用公共列，则公共列不能加表别名  
    6. 交叉连接 cross join，笛卡尔乘积 Cartesian Products  
    7. 连接的方式  
        # sort_merge_join  
        # hash_join  
        # nest_loop

知识点–语法：USING，ON，指定连接条件

    1. 连接条件  
        # 作为连接条件的两个列，可以具有不同的数据类型，但必须可以相互进行隐式转换  
    2. USING  
        # 使用公共列自动进行等值连接，并会删除右表的公共列，将左表的公共列作为表的第一列  
        # 语法
            - USING (<column_name>[, ...])
    3. ON  
        # ON 方式会保留公共列，重名的公共列会重新起个名  
        # ON 方式是功能最全面的连接方式，能完成所有连接要求  
        # ON 方式最好使用表别名
        # 非等值连接只能使用 ON 方式  
        # 语法
            - ON (<join_condition> [{AND|OR} <join_condition>][ ...])

知识点–语法：连接 JOIN

    1. 自然连接 NATURAL JOIN  
        # FROM <table_a> NATURAL [INNER] JOIN <table_b>  
        # FROM <table_a> NATURAL {FULL|LEFT|RIGHT} [OUTER] JOIN <table_b>  
    2. 内连接 INNER JOIN  
        # FROM <table_a> [INNER] JOIN <table_b> USING (<column_name> [, ...])  
        # FROM <table_a> [INNER] JOIN <table_b> ON <condition>  
    3. 左外连接 LEFT OUTER JOIN  
        # FROM <table_a> LEFT [OUTER] JOIN <table_b> ON <condition>  
        # FROM <table_a> LEFT [OUTER] JOIN <table_b> USING (<column_name> [, ...])  
    4. 右外连接 RIGHT OUTER JOIN  
        # FROM <table_a> RIGHT [OUTER] JOIN <table_b> ON <condition>  
        # FROM <table_a> RIGHT [OUTER] JOIN <table_b> USING (<column_name> [, ...])  
    5. 全外连接 FULL OUTER JOIN  
        # FROM <table_a> FULL [OUTER] JOIN <table_b> ON <condition>  
        # FROM <table_a> FULL [OUTER] JOIN <table_b> USING (<column_name> [, ...])  
    6. 笛卡尔乘积  
        # FROM <table_a> , <table_b> ...  
        # FROM <table_a> CROSS JOIN <table_b> ...

知识点–概念：连接旧语法

    1. 等值连接，FROM table_A, table_B WHERE A_col = B_col  
    2. 左外连接，FROM table_A, table_B WHERE A_col = B_col(+)  
    3. 右外连接，FROM table_A, table_B WHERE A_col(+) = A_col  
    4. 笛卡尔乘积，FROM table_A, table_B, ...  
    5. 旧语法和新语法最好不要混用，有些情况下会报错

1.1.10 子查询

知识点–概念：子查询 subquery

    1. 介绍
        # 子查询，也称为内查询  
        # 父查询，也称为外查询、主查询  
        # 先执行子查询，后执行父查询  
    2. 子查询可以没有返回行  
    3. 子查询的嵌套和解嵌套  
        # 多数情况下，优化器会自动对子查询进行解嵌套，将子查询融合到主查询中  
        # 少数情况下，子查询不会被解嵌套，此时，对于主查询的每一个返回行，都会进行一次子查询，严重影响性能，包括以下情况
            * 层级查询  
            * 关联子查询  
            * 子查询中包含 ROWNUM  
            * 子查询包含集合运算符  
            * 子查询包含聚合函数  
    4. 内联视图 inline_view  
        # 也叫嵌入式视图  
            * 在 FROM 中使用子查询，将子查询的结果当做一个表来用  
            * 在 <select_list>、<condition> 上使用单行子查询，作为一个标量使用  
    5.单行子查询 single_row_subquery  
        # 返回值为一行一列的子查询  
        # 也称为标量子查询，因为 oralce 会将单行子查询当做一个标量进行处理  
        # 如果单行子查询没有返回行，则视为返回 NULL 值

知识点–概念：关联子查询 correlated subquery

    1. 介绍
        # 在子查询中，需要使用父查询的值作为 WHERE 条件，就称为关联子查询
        # 通常需要在父查询中定义表别名，在子查询中使用表别名  
        # 关联子查询，对主查询的每一行，都执行一次子查询  
            - 因而关联子查询的效率奇差
            - 建议使用连接进行替代  
                关联子查询次数多，占用空间少，属于时间换空间
                连接循环次数少，占用空间多，属于空间换时间
    2. 范例  
        # 查询每个部门的工资的前三名  
            - SELECT * FROM employees t_f WHERE employee_id IN  
                (SELECT employee_id FROM employees  
                    WHERE department_id = t_f.department_id AND salary IS NOT NULL  
                    ORDER BY salary DESC  
                    FETCH FIRST 3 ROWS WITH TIES )  
                ORDER BY department_id, salary DESC  
                ;

1.1.11 组合查询

知识点–概念：集合运算符，组合查询

    1. 集合运算符 Set Operators  
        # 实现数学上的集合运算  
            - UNION                <-- 并集  
            - UNION ALL            <-- 并集，不去重，不排序  
            - INTERSECT            <-- 交集  
            - MINUS                <-- 差集  
    2. 集合运算符的去重、保留 NULL、自动排序  
        # UNION ALL  
            - 不去重  
            - 不自动排序  
            - 保留 NULL  
        # UNION、INTERSECT、MINUS  
            - 去重  
            - 自动排序  
                以第一个查询的第一列为排序依据，升序排列  
            - 保留 NULL，对于多个 null ，合并为一个 null  
    3. 集合运算符和 ORDERY BY  
        # 每个查询不允许添加自己 ORDER BY  
        # 只能在总查询的最后添加总 ORDER BY  
    4. 集合运算符的优先级  
        # 集合运算符都具有相同的优先级  
        # 一个查询中出现多个集合运算符，按照从前向后的顺序，依次执行  
        # 括号可以改变集合运算符的优先级  
    5. 组合查询 Compound Query    
        # 使用集合运算符的查询称为组合查询  
        # 组合查询的 column  
            - 列以第一个查询为基础，不管其他的查询是否加了括号  
            - 所有其他查询的列的数量，要与第一个查询一致  
        # 组合查询的 column_name  
            - 列名以第一个查询为基础，不管其他的查询是否加了括号  
            - 其他所有查询的列名被忽略  
        # 组合查询的 data type  
            - 数据类型以第一个查询的数据类型的 data_type_group 为基础，不管其他的查询是否加了括号  
            - 其他所有查询的列的数据类型，要与第一个查询的对应的列，属于同一个 data_type_group  
            - 当数据类型为属于同一个 data_type_group 的不同数据类型，或宽度不同，则会进行数据类型的转换  
                如果两个查询都是 char 型，并且宽度相同，则结果为 char 型，具有相同宽度  
                如果两个查询都是 char 型，具有不同宽度，则结果为 varchar2 型，具有较宽的那个宽度  
                如果有任意一个为 varchar2 型，则结果为 varchar2 型

1.3 层级查询

1.3.1 层级查询

知识点–概念：层级查询 hierarchical query

    1. 介绍  
        # 查询数据中所包含的树形结构逻辑  
        # 结果是一个倒置树 inverted tree  
    2. 四种节点  
        # 根节点 root  
        # 父节点 parent  
        # 子节点 child  
        # 叶节点 leaf  
    3. 层级查询，需要表中有一列作为记录树形结构的逻辑关系，PRIOR 运算符  
        # 必须保证每一个节点都是唯一的，不能出现重复节点，否则会出现重复树  
        # 必须保证每一个子节点只有一个父节点，否则报错  
        # 例如 hr.employees 中的 manager_id 列  
        # 使用 PRIOR 运算符在 connect_by_clayse 中，来指定父子关系  
    4. 层级查询子句，放在 where clause 和 group clause 之间  
    5. 层级查询执行，在 join 之后执行，在 where clause 之前执行  
    6. 不建议在层级查询中使用 order by 和 group by，因为它们会冲掉最终的层级结果，不过可以使用 ORDER SIBLINGS BY

1.3.2 层级查询语法

知识点–语法：CONNECT BY 层级查询

    1. 语法  
        # SELECT ... FROM ... WHERE ... CONNECT BY [NOCYCLE] <connect_by_condition> [START WITH <start_condition>] ORDER BY ... ;  
        # SELECT ... FROM ... WHERE ... START WITH <start_condition> CONNECT BY [NOCYCLE] <connect_by_condition> ORCER BY ... ;  
            - NOCYCLE  
                显示环回的行，而不是报错  
                环回表示，一个节点，同时是另一个节点的祖先和后代  
            - CONNECT BY <connect_by_condition>  
                指定父子关系，确定树的逻辑关系  
            - START WITH <start_condition>  
                指定根节点，用来限定是查询一整棵树还是一棵分支  
                如果省略，则会返回所有的树，亦即所有子树都会被当成一个单独的树重复写一遍  
    2. <start_condition>  
        # 用来指定根节点的 child_item 项的性质，例如 employee_id = 100  
    3. <connect_by_condition>  
        # PRIOR 运算符  
            - 一元运算符，只能用在层级查询的 <connect_by_condition> 中，用来指定父节点项，以确定父子关系  
            - <connect_by_condition> 中必须有 PRIOR  
            - <connect_by_condition> 可以包含多个条件，但只能有一个条件需要指定 PRIOR，although you can have multiple PRIOR conditions  
            - PRIOR 放在等号左边或右边都可以  
            - PRIOR 一般都是指定的是 =，也可以使用其他运算符如比较运算符，但容易产生环回  
            - 格式  
                PRIOR <parent_item> = <child_item>  
                <child_item> = PRIOR <parent_item>  
            - 范例  
                PRIOR employee_id = manager_id            -- 子节点的 manager_id 指向父节点的 employee_id  
    4. <connect_by_condition> 和 <where_condition>  
        # <connect_by_condition> 先于 <where_condition>  
        # <connect_by_condition> 是在树的产生之前进行筛选，不满足条件的节点，连同其子孙，整个树枝都会被减掉  
        # <where_condition> 是在树的产生之后进行筛选，不满足条件的节点被删除，但子孙不受影响，因而会产生不连续的树  
    5. 层级查询的执行过程  
        # 通过 START WITH <start_condition>，确定根节点  
        # 找到根节点的子节点  
        # 递归找寻每个子节点的子节点，最终确立整个倒置树

知识点–语法：ORDER SIBLINGS BY 层级查询排序

    1. 介绍  
        # 层级查询排序，只能出现在层级查询中  
        # 按父节点进行分组  
        # 只对属于同一个父节点的所有子节点进行排序

1.3.3 层级查询相关元素

知识点–概念：CONNECT_BY_ROOT 运算符，返回根节点的某列的值

    1. 介绍  
        # 单目运算符，只能用在层级查询中  
        # 如果层级查询没有指定根节点，导致层级查询中有多棵树，则可以用 CONNECT_BY_ROOT 可以用来区分不同的树  
        # 可以放在 <select_list> 或 WHERE 中，不能用在 START WITH 和 CONNECT BY 中  
    2. 用法  
        # SELECT CONNECT_BY_ROOT <column_name> FROM <talbe_name> <hierarchical_query_clause> ;  
        # SELECT <expr_1>, <expr_2> FROM <table_name> WHERE CONNECT_BY_ROOT <column_name> = '321' <hierarchical_query_clause> ;

知识点–概念：CONNECT_BY_ISCYCLE，层级查询伪列，检查是否出现环回

1.4.1 时间有效性查询

知识点–概念：时间有效性查询 temporal validity

    1. 时间有效性  
        # 在表中储存数据的有效时间段  
        # 查询时，需要指明有效时间段  
        # 仅返回符合有效时间段的行  
        # ILM information lifecycle mamagement  
    2. 时间有效性字段 valid_time_column  
        # 自动设置为隐藏列形式  
        # 用来作为进行时间有效性判断的依据，储存有效时间的时间段  
        # 实际会作为两个列进行储存数据，<valid_time_column_name>_START，<valid_time_column_name>_END，标识有效时间的时间段  
            - 可以显示指明 START 列和 END 列  
            - 也可以仅指明 valid_time_column_name，由 oracle 隐式自动转换为 START 列和 END 列  
            - CREATE TABLE 语句中，使用 PERIOD FOR 从句指明时间有效性字段  
    3. 时间有效性查询的两种方式  
        # 基于时间点的查询

            - 返回给定时间点位于有效时间段内的行

        # 基于时间段的查询  
            - 查询时给出时间段  
            - 返回给定时间段与有效时间段有重合的行  
            - SELECT 语句中，使用 VERSIONS BETWEED PERIOD FOR 实现  
    4. DBMS_FLASHBACK_ARCHIVE.ENABLE_AT_VALID_TIME  
        # 在 session 级指定有效时间点  
        # 之后所有的查询和 DML 自动进行对有效时间的基于时间点的查询  
        # 注意参数 ASOF 和 CURRENT 要大写，否则过程不会生效，并且不会给出错误提示

知识点–语法：时间有效性查询语法

3.4.1 MERGE

    1.作用  
        # 将 <source_table> 的内容，融合到 <target_table> 中  
            - 批量进行 INSERT 和 UPDATE  
            - <target_table> 中存在的行，进行 UPDATE  
            - <target_table> 中不存在的行，进行 INSERT  
    2.MERGE 的优点  
        # 节约资源的申请和消耗  
    3.语法  
        # MERGE INTO <target_table> USING <source_table>  
            ON(<condition>)  
            WHEN MATCHED THEN  
                UPDATE SET {<column>=<value>}[, ...]  
                [<where_clause>]  
                [DELETE <where_clause>]  
            WHEN NOT MATCHED THEN  
                INSERT [(<column>[, ...])] VALUES (<value>[, ...])  
                [where_clause]  
            ;  
        # WHEN MATEHED 和 WHEN NOT METCH 的顺序不能乱，但可以只出现其中的某一个  
        # DELETE <where_clause>  
            - 在 UPDATE 时，同时删除 <target_table> 中的某些行  
    4.范例  
        # CREATE TABLE testemp AS SELECT employee_id, first_name, salary FROM hr.employees ;

            ON ( testemp.employee_id = employees.employee_id )

                UPDATE SET testemp.first_name = employees.first_name || ' ' || employees.last_name

6.4.3 UNIQUE 约束和 PRIMARY KEY 约束

知识点–概念：UNIQUE 约束

    1.作用  
        # 保证不会有两行相同的记录  
        # 如果 UNIQUE 约束定义在一列上，则这一列不会有两个相同的记录  
        # 如果 UNIQUE 约束定义在多列上，则这几列不会有两行完全相同的记录  
            - 两行的每一列的数据都相同，违反约束  
            - 两行中，部分列是 NULL ，部分列不是 NULL，则所有非 NULL 的列数据相同，就违反约束  
            - 两行所有列的数据都是 NULL，不违反约束  
    2.唯一键 unique key 和组合唯一键 composite unique key  
        # UNIQUE 约束中，所包含的列，称为唯一键 unique key  
        # UNIQUE 约束中，如果包含多个列，则所有的这些列的组合称为一个组合唯一键 composite unique key  
    3.UNIQUE 约束和 UNIQUE INDEX  
        # ORACLE 隐式的为每一个 UNIQUE 约束创建一个 UNIQUE INDEX  
        # UNIQUE INDEX 的名字与约束名相同  
        # 可以添加 <using_index> 来显示指定使用的索引  
    4.UNIQUE 约束在 CREATE TABLE 语句中的位置  
        # 既可以写在列级约束中，也可以写在表级约束中  
        # 如果涉及多个列，则只能写在表级约束中  
        # 范例  
            - CREATE TABLE test ( a NUMBER UNIQUE ) ;  
            - CREATE TABLE test ( a NUMBER, b NUMBER, UNIQUE ( a, b ) ) ;

知识点–概念：PRIMARY KEY，主键约束

    1.作用  
        # 用来唯一确定一个表中的某一行  
        # 一个表只能创建一个 PRIMARY KEY 约束  
        # 相当于 NOT NULL 和 UNIQUE 的组合  
        # 可以只包含一列，也可以包含多列  
        # 如果一个 PRIMARY KEY 包含多个列，则每列都不允许插入 NULL  
    2.PRIMARY KEY 约束和 UNIQUE INDEX  
        # ORACLE 隐式的为每一个 PRIMARY KEY 约束创建一个 UNIQUE INDEX  
        # UNIQUE INDEX 的名字与主键的名字相同  
        # 可以添加 <using_index> 来显示指定使用的索引  
    3.组合主键 composite primary key  
        # 如果一个主键约束中包含多个列，则称为组合主键 composite primary key  
    4.PRIMARY KEY 约束在 CREATE TABLE 语句中的位置  
        # 既可以写在列级约束中，也可以写在表级约束中  
        # 如果涉及多个列，则只能写在表级约束中  
        # 范例  
            - CREATE TABLE test ( a NUMBER PRIMARY KEY) ;  
            - CREATE TABLE test ( a NUMBER PRIMARY KEY, b NUMBER) ;

        # USING INDEX (<create_index_statement>)  
    2.范例

6.4.4 外键约束

知识点–概念：FOREIGN KEY，外键约束

    1.作用  
        # 建立两个表之间的多对一关系，子表为多，父表为一  
        # 定义在子表上，引用父表中的内容  
        # 同一个列可以建立多个 FOREIGN KEY 约束  
        # 一个 FOREIGN KEY 约束只能引用一个父表  
        # 外键约束包含多个列  
            - 父表的 PRIMARY KEY 约束或 UNIQUE 约束必须是包含多个列的约束  
            - 子表外键中的列的数量必须与父表的 PRIMARY KEY 约束或 UNIQUE 约束列的数量相同  
    2.referenced_column  
        # 父表中的被引用列  
        # referenced_column 必须已经建立了 PRIMARY KEY 约束或 UNIQUE 约束  
        # 外键实际上是指向父表的 PRIMARY KEY 约束或 UNIQUE 约束  
    3.referencing_column  
        # 子表中引用父表的列，称为 referencing_column  
        # referencing_column 如果没有 NOT NULL 约束，则可以插入空值，不论父表的 referenced_column 是否设定了 NOT NULL  
    4.不能随意对父表进行删除，除非相关的删除语句中加上了相应的关键字  
        # 无法对父表 DROP TABLE，即使子表为空表  
        # 无法对父表 ALTER TABLE DROP (<referenced_column>)，即使子表为空表  
        # 无法对父表 DELETE row，除非此行在 referenced_column 上的值没有被子表使用，或者 FOREIGN KEY 设置了 ON DELETE SET NULL 或 ON DELETE CASCADE  
        # 无法删除父表的 PRIMARY KEY 或 UNIQUE KEY，除非指定了 CASCADE  
            - 子表的外键被级联删除  
            - 子表的数据会保留，不论是否设置了 ON DELETE SET NULL 或 ON DELETE CASCADE  
    5.语法  
        # 列级，[CONSTRAINT <constraint_name>] REFERENCES <object_name>(<column>) [ON DELETE {CASCADE | SET NULL}]  
        # 表级，[CONSTRAINT <constraint_name>] FOREIGN KEY (<column>[, ...]) REFERENCES <object_name>(<column>[, ...]) [ON DELETE {CASCADE | SET NULL}]  
        # ON DELETE  
            - ON DELETE CASCADE，删除父表中的行时，子表中引用了相应值的行会被一起删除  
            - ON DELETE SET NULL，删除父表中的行时，子表中引用了相应值的行会被设置为 NULL  
            - 省略 ON DELETE 部分，表示 no action，即若子表中引用了相应的值，则父表中相应的行不能被 DELETE row  
    6.FOREIGN KEY 约束在 CREATE TABLE 语句中的位置  
        # 既可以写在列级约束中，也可以写在表级约束中  
        # 如果涉及多个列，则只能写在表级  
        # 写在列级约束时，可以省略此列的数据类型  
        # 列级范例  
            - CREATE TABLE test ( a REFERENCES employees ( employee_id ) ) ;  
            - CREATE TABLE test ( a NUMBER REFERENCES employees ( employee_id ) ) ;  
        # 表级范例  
            - CREATE TABLE test ( a NUMBER, b NUMBER, FOREIGN KEY ( a ) REFERENCES employees ( employee_id )) ;

知识点–操作：查询外键的父表子表关系

    1.给出子表，看其有哪些父表  
        # &owner                -- 指定子表的 owner  
        # &child_table_name     -- 指定子表的表名  
        # SELECT  
            child.constraint_type "child_cons_type",  
            child.table_name "child_table",  
            child.constraint_name "child_cons",  
            father.constraint_type "father_cons_type",  
            father.table_name "father_table",  
            father.constraint_name "father_cons"  
          FROM  
            dba_constraints child LEFT JOIN dba_constraints father  
            ON child.r_owner = father.owner AND child.r_constraint_name = father.constraint_name  
          WHERE  
            child.owner = upper('&owner')  
            AND child.table_name = upper('&child_table_name')  
            AND child.constraint_type = 'R'  
         ;  
    2.给出父表，看其有哪些子表  
        # &owner                -- 指定父表的 owner  
        # &child_table_name     -- 指定父表的表名  
        # SELECT  
            child.constraint_type "child_cons_type",  
            child.table_name "child_table",  
            child.constraint_name "child_cons",  
            father.constraint_type "father_cons_type",  
            father.table_name "father_table",  
            father.constraint_name "father_cons"  
          FROM dba_constraints father LEFT JOIN dba_constraints child  
            ON child.r_owner = father.owner AND child.r_constraint_name = father.constraint_name  
          WHERE father.owner = upper('&owner')  
            AND father.table_name = upper('&child_table_name')  
            AND child.constraint_type = 'R'  
         ;

6.4.5 约束的状态

知识点–概念：启用和禁用约束，ENABLE，DISABLE

    1.ENABLE  
        # 对于之后的 DML 操作，进行约束的检查  
    2.DISABLE  
        # 对于之后的 DML 操作，不进行约束的检查  
        # 若 PRIMARY KEY 和 UNIQUE 约束被外键所引用，需要指定 DISABLE CASCADE 来级联禁用约束  
        # 禁用 PRIMARY KEY 和 UNIQUE 约束会自动删除索引，重新启用约束会自动建立索引  
        # 对于指定 DISABLE CASCADE 被级联禁用的外键，重新启用约束无法级联启用外键  
    3.ENABLE 和 DISABLE 的创建与修改  
        # 可以在 CREATE TABLE 时指定  
        # 可以在 ALTER TABLE ADD CONSTRAINT 中指定  
        # 可以通过 ALTER TABLE MODIFY 进行修改  
        # 位置，放在 DEFERRABLE 部分之后

知识点–概念：约束的生效和失效，VALIDATE，NOVALIDATE

    1.VALIDATE  
        # 对于表中已经存在的数据，进行约束的检查  
    2.NOVALIDATE  
        # 对于表中已经存在的数据，不进行约束的检查  
    3.VALIDATE 和 NOVALIDATE 的创建与修改  
        # 可以在 CREATE TABLE 时指定  
        # 可以在 ALTER TABLE ADD CONSTRAINT 中指定  
        # 可以通过 ALTER TABLE MODIFY 进行修改  
        # 位置，放在 ENABLE、DISABLE 之后  
    4.生效和失效的应用  
        # 一个常见的应用场景  
            - DISABLE 约束并 NOVALIDATE 约束  
            - INSERT 新数据，UPDATE 新数据  
            - 提交  
            - VALIDATE 约束，如果失败，则对违反约束的数据进行修改，直到约束可以生效  
            - ENABLE 约束

6.4.6 延迟约束

知识点–概念：延迟约束 DEFERRABILITY

    1.介绍  
        # 修改在事务中对数据进行约束的检查的时间点  
        # NOT DEFERRABLE  
            - 普通约束，非延迟约束  
            - 不允许修改约束的检查时间点  
        # DEFERRABLE  
            - 延迟约束  
            - 允许修改约束的检查时间点  
            - IMMDEIATE，在语句结束时进行约束的检查，效果与普通约束相同  
            - DEFERRED，在事务结束时进行约束的检查  
    2.延迟约束的初始状态  
        # INITIALLY IMMDEIATE  
        # INITIALLY DEFERRED  
        # 指定约束的初始状态  
            - 可以在 CREATE TABLE 中新建约束时指定  
            - 可以在 ALTER TABLE ADD CONSTRAINT 中新建约束时指定  
            - 可以用 ALTER TABLE MODIFY CONSTRAINT 来修改延迟约束的初始状态  
            - 不能用 ALTER TABLE MODIFY CONSTRAINT 将非延迟约束修改为延迟约束，如果想修改只能删除约束后重建  
            - 位置，位于 constraint_contents 之后  
    3.延迟约束的优点  
        # 提供了一种可以在事务之内启用和禁用约束的手段  
    4.应用场景  
        # 延迟约束是为 SET CONSTRAINT 语句专门提供的一个特性  
        # 可以在添加大量低质量的数据时，提高成功率  
        # 主要用在物化视图的场景  
        # 一个应用场景  
            - SET CONSTRAINT ALL DEFERRED  
            - 执行添加数据的 DML 语句  
            - SET CONSTRAINT <constraint_1> DEFERRED，如果失败，检查并修改引起失败的数据，直到语句成功  
            - SET CONSTRAINT <constraint_2> DEFERRED，如果失败，和上一步的操作相同，直到所有的约束全部通过检查  
            - COMMIT

知识点–语法：SET CONSTRAINT，修改延迟约束的检查时间

    1.介绍  
        # SET CONSTRAINT 的设置会覆盖延迟约束的初始状态  
        # SET CONSTRAINT 的修改，限定在一个事务之内  
        # 可以使用 ALTER SESSION SET CONSTRAINT 的方式，作用相同  
    2.SET CONSTRAINT IMMEDIATE 语句如果失败，不会回退已经执行的 DML 操作  
    3.SET CONSTRAINT 语法  
        # SET {CONSTRAINT|CONSTRAINTS} ALL {IMMEDIATE|DEFERRED} ;  
        # SET {CONSTRAINT|CONSTRAINTS} <constraint_name>[, ...] {IMMEDIATE|DEFERRED} ;  
            - CONSTRAINT|CONSTRAINTS        -- 语法提示词  
            - ALL，修改所有延迟约束  
            - DEFERRED，将延迟约束的检查时间修改到事务提交前，相当于临时禁用约束  
            - IMMEDIATE，将延迟约束的检查时间修改到语句结束前，相当于重新启用约束  
    4.ALTER SESSION 语法  
        # ALTER SESSION SET {CONSTRAINT|CONSTRAINTS} ALL {IMMEDIATE|DEFERRED} ;  
        # ALTER SESSION SET {CONSTRAINT|CONSTRAINTS} <constraint_name>[, ...] {IMMEDIATE|DEFERRED} ;

6.4.7 CREATE TABLE 添加约束语句

知识点–语法：CREATE TABLE 创建表时创建约束

    1.语法  
        # CREATE TABLE <table_name> ({<column_name> <data_type> [DEFAULT <value>] [<column_constraint>[ ...]]}[, ...] [, <table_constraint>[, ...] ]) ;  
        # <column_constraint>  
            - [CONSTRAINT <constraint_name>] PRIMARY KEY  
            - [CONSTRAINT <constraint_name>] UNQIUE  
            - [CONSTRAINT <constraint_name>] NOT NULL  
            - [CONSTRAINT <constraint_name>] REFERENCES <table_name> (<column_name>)  
            - [CONSTRAINT <constraint_name>] CHECK (condition)  
        # <table_constraint>  
            - [CONSTRAINT <constraint_name>] PRIMARY KEY (<column_name>[, ...])  
            - [CONSTRAINT <constraint_name>] UNQIUE (<column_name>[, ...])  
            - [CONSTRAINT <constraint_name>] FOREIGN KEY (<column_name>[, ...]) REFERENCES <table_name> (<column_name>[, ...])  
            - [CONSTRAINT <constraint_name>] CHECK (condition)

6.4.8 ALTER TABLE 修改约束语句

知识点–语法：ALTER TABLE 添加删除约束

    1.添加列级约束  
        # ALTER TABLE <table_name> MODIFY ( <column_name> [CONSTRAINT <column_constraint>] ) ;                  <-- 添加列级约束  
            - ALTER TABLE <table_name> MODIFY ( <column_name> [CONSTRAINT <constraint_name>] NOT NULL) ;            <-- 添加 NOT NULL 约束  
            - ALTER TABLE <table_name> MODIFY ( <column_name> NULL) ;                                               <-- 删除 NOT NULL 约束  
            - NOT NULL 约束只能使用 ALTER TABLE MODIFY 语句添加和删除  
            - 或者使用 CHECK 约束来实现 NOT NULL 约束  
        # ALTER TABLE <table_name> ADD [CONSTRAINT <constraint_name>] <table_constraint> ;                      <-- 添加表级约束  
    2.删除约束  
        # ALTER TABLE <table_name> DROP CONSTRAINT <constraint_name> [CASCADE] [ONLINE] ;                           <-- 删除约束  
            - CASCADE， 同时删除依赖此约束的其他外键约束  
            - ONLINE，允许在删除约束的行为结束前，执行 DML 操作  
        # ALTER TABLE <table_name> DROP PRIMARY KEY [CASCADE] [{KEEP|DROP} INDEX] [ONLINE] ;                        <-- 删除主键  
        # ALTER TABLE <table_name> DROP UNIQUE (<column_name> [, ...]) [CASCADE] [{KEEP|DROP} INDEX] [ONLINE] ;     <-- 删除唯一约束

知识点–语法：ALTER TABLE MODIFY CONSTRAINT 修改约束

    1. 重命名约束  
        # ALTER TABLE <table_name> RENAME CONSTRAINT <old_name> TO <new_name> ;                         <-- 约束重命名  
    2. 启用和禁用约束  
        # ALTER TABLE <table_name> ENABLE CONSTRAINT <constraint_name> ;                                <-- 启用约束  
        # ALTER TABLE <table_name> DISABLE CONSTRAINT <constraint_name> [CASCADE] ;                     <-- 禁用约束  
            - CASCADE  
                仅在 DISABLE PRIMARY KEY 和 UNIQUE 约束，并且是被其他外键所引用时，才可以使用  
                用于级联禁用相应的外键  
    3. 修改约束的状态  
        # ALTER TABLE <table_name> MODIFY CONSTRAINT <constraint_name> <constraint_state> [CASCADE] ;       <-- 修改约束状态或检查时间点  
        # ALTER TABLE <table_name> MODIFY PRIMARY KEY <constraint_state> [CASCADE] ;                        <-- 修改主键的状态或检查时间点  
        # ALTER TABLE <table_name> MODIFY UNIQUE (<column_name> [, ...] ) <constraint_state> [CASCADE] ;    <-- 修改唯一约束的状态或检查时间点  
        # <constraint_state>  
            - [ENABLE|DISABLE] {VALIDATE|NOVALIDATE}                                            <-- 非延迟约束  
            - INITIALLY {IMMEDIATE|DEFERRED} [ENABLE|DISABLE] [VALIDATE|NOVALIDATE]             <-- 延迟约束

知识点–概念：只读表

            - FLASHBACK TABLE

    2.只读表可以进行的操作  
        # 删除表  
            - DROP TABLE  
        # 不修改数据的 DDL

            - ALTER TABLE MOVE  
            - ALTER TABLE ENABLE ROW MOVEMENT  
            - ALTER TABLE SHRINK  
            - ALTER TABLE ADD/DROP SUPPLEMENTAL LOG  
        # 不修改数据的列 DDL  
            - ALTER TABLE ADD COLUMN  
            - ALTER TABLE MODIFY COLUMN  
            - ALTER TABLE DROP UNUSED COLUMNS  
        # 添加修改删除约束  
            - ALTER TABLE ADD/MODIFY/DROP/ENABLE/DISABLE CONSTRAINT  
        # 可以对相应的索引进行操作

            - ALTER INDEX  
            - DROP INDEX

    1.语法

7.2.1 隐藏列

知识点–概念：隐藏列 INIVISIBLE

    1.介绍  
        # 12C 新功能  
        # 可以用 SELECT <column_name> 查看  
        # 无法用 DESC 命令查看  
        # 无法用 SELECT * 查看  
        # INSERT 语句要显式指定 <column_name>  
        # %ROWTYPE 会自动忽略掉隐藏列  
    2.相关的字典记录  
        # DBA_TAB_COLS 的 HIDDEN_COLUMN 列，记录是否是隐藏列  
        # DBA_TAB_COLS 的 COLUMN_ID 列会被置 NULL  
    3.虚拟列可以设置为隐藏列  
    4.隐藏列的限制  
        # 外部表，簇表，临时表，不可以设置隐藏列  
        # You cannot make a system-generated hidden column visible.  
    5.应用  
        # 修改列的顺序  
            - 设置一个列 INVISIBLE 后再 VISIABLE，则列会排列到表的最末尾  
    6.语法  
        # 关键字放在 DEFALUT 之前  
        # CREATE TABLE <table_name> ( <column_name> <data_type> [INVISIBLE|VISIBLE] )  
        # ALTER TABLE <table_name> MODIFY (<column_name> {INVISIBLE|VISIBLE})

7.2.2 虚拟列

知识点–概念：虚拟列 VIRTUAL

    1.虚拟列  
        # 通过引用同一个表的其他列同一行的数据，来计算而生成数据，不实际储存数据  
        # 只能在堆表上创建  
        # 不能引用其他虚拟列  
        # 不能引用其他的表的列  
        # 不能被 INSERT，不能被 UPDATE  
        # 可以省略数据类型  
        # 可以被 select，可以创建索引，可以用在 where 中  
        # 可以创建约束  
        # 可以被外键引用  
    2.相关的字典记录  
        # DBA_TAB_COLS 的 VIRTUAL_COLUMN 列，记录是否是隐藏列  
    3.创建虚拟列语法  
        # 用在 CREATE TABLE、ALTER TABLE ADD 中创建和添加虚拟列  
        # <column_name> [<data_type>] [VISIBLE|INVISIBLE] [GENERATED ALWAYS] AS (<column_expression>) [VIRTUAL] [<constraints>]  
            - AS (<column_expression>)        -- 指定虚拟列的值的表达式  
            - GENERATED ALWAYS                -- 语法提示词  
            - VIRTUAL                        -- 语法提示词  
    4.修改虚拟列语法  
        # 用在 ALTER TABLE MODIFY 修改虚拟列  
        # <column_name> [<data_type>] [GENERATED ALWAYS] AS (<column_expression>) [VIRTUAL]

7.2.3 自增列

知识点–概念：自增列 IDENTITY PROPERTY

    1.自增列  
        # 12C 新功能  
        # oracle 会自动创建一个序列  
        # 对每一次 INSERT 语句，自动将序列的值添加到自增列中  
        # ALWAYS 型的自增列不能显式 INSERT  
        # 序列可以是递增的，也可以是递减的  
        # 自增列会隐式的建立一个 NOR DEFERRABLE 的 NOT NULL 约束  
    2.自增列的作用  
        # 为表生成不会产生中断的数据，作为主键  
        # 如果自增列使用了 CACHE，那么如果关闭数据库，则还是可能产生中断的数据  
    3.自增列的限制  
        # 取代 DEFAULT 的设置  
        # 自增列的数据类型必须是 NUMBER  
        # 一个表只能有一个自增列  
        # 子查询建表不会继承自增列  
    4.相关的字典记录  
        # DBA_TAB_COLS 的 IDENTITY_COLUMN 列，记录是否是自增列  
    5.已存在数据的表添加自增列  
        # ALTER TABLE ADD 方式添加一个自增列  
        # ORACLE 会自动在已存在的行上随机填充 id  
        # 对之后的 INSERT 则正常进行标识  
    6.创建自增列语法  
        # CREATE TABLE，ALTER TABLE ADD，ALTER TABLE MODIFY  
        # 位置，原来 DEFAULT 的位置  
        # GENERATED ALWAYS AS IDENTITY [(<identity_option>[ ...])]  
        # GENERATED BY DEFAULT [ON NULL] AS IDENTITY [(<identity_option>[ ...])]  
            - ALWAYS            -- 自增列的数据只能由 oralce 根据序列来填充，不能手动 INSERT，或者在 INSERT 语句中使用 DEFAULT 关键字  
            - BY DEFAULT        -- 自增列的数据可以显式指定  
            - ON NULL            -- 如果 INSERT 语句在自增列上的定义为 NULL，则填充序列  
        # <identity_options>  
            - 设置序列的属性，所有内容与 CREATE SEQUENCE 相同  
            - START WITH (<N>|LIMIT VALUE)  
            - INCREMENT BY <N>  
            - MAXVALUE <N> | NOMAXVALUE  
            - MINVALUE <N> | NOMINVALUE  
            - CYCLE | NOCYCLE  
            - CACHE <N> | NOCACHE  
            - ORDER | NOORDER  
    7.删除自增列语法  
        # ALTER TABLE MODIFY ( <column_name> DROP IDENTIFY)  
            - 删除自增列的自增属性  
            - 级联删除相应的序列  
            - 已存在的数据不受影响  
    8.START WITH LIMIT VALUE  
        # 只能用在 ALTER TABLE MODIFY 语句中，是 <identity_options> 中唯一一个与 CREATE SEQUENCE 不同的语句  
        # 作用  
            - 将已存在数据的列修改为自增列时，会自动选取已存在数据中的最值，之后的序列的起点从最值开始  
            - 递增型则最大值，递减型则最小值  
        # 例子  
            - ALTER TABLE <table_name> MODIFY (<column_name> GENERATED ALWAYS AS IDENTITY (START WITH LIMIT VALUE ))

7.4 外部表

7.4.1 外部表

知识点–概念：外部表 EXTERNAL TABLE

    1.外部表  
        # 直接将 OS 上的文本文件或 EXCEL 文件等，当做表的方式进行读取  
        # 本质上是一个指向 OS 数据文件的指针  
        # 需要指定目录对象  
        # 外部表只支持 READ  
        # 不支持 DML  
        # 不支持建立索引  
    2.外部表的接口方式 ACCESS DRIVER  
        # ORACLE_LOADER  
            - sql_loader 方式  
            - 默认方式  
            - 支持文本文件、csv 文件、excel 文件  
        # ORACLE_DATAPUMP  
            - datapump 方式  
            - 仅支持 dmp 文件，并且是使用 datapump 创建的 dmp 文件，不支持 exp 创建的 dmp 文件  
        # ORACLE_HDFS  
        # ORACLE_HIVE  
    3.外部表的优点  
        # 通过外部表，可以直接从外部数据文件中加载数据，更简捷，不需要中转表，这对处理大量数据可以节省空间  
        # sql_loader 方式，可以直接读取文本文件  
        # datapump 方式，可以直接从 dmp 文件中调取数据，也可以在数据库加载 dmp 文件前通过使用外部表来验证 dmp 文件中的数据  
    4.卸载数据和装载数据  
        # 卸载数据，从数据库中将数据保存到外部数据文件作为外部表  
        # 装载数据，从外部表中将数据导入到数据库中  
            - INSERT 配合子查询方式  
            - CREATE TABLE 配合子查询方式  
    5.字典  
        # DBA_EXTERNAL_TABLES        -- 记录所有的外部表  
        # DBA_EXTERNAL_LOCATIONS    -- 记录外部表指向的操作系统文件

7.4.2 创建外部表

知识点–语法：CREATE TABLE 创建外部表

    1.ORACLE_LOADER 装载数据  
        # CREATE TABLE <table_name> (<column_list>)            <== 外部表的的列与操作系统文件中的列，列的数量要对应，列的数据类型要对应  
            ORGANIZATION EXTERNAL(  
                TYPE ORACLE_LOADER                            <== 指定接口为 sql_loader  
                DEFAULT DIRECTORY <dir_name>                <== 指定目录对象  
                ACCESS PARAMETERS(                            <== 指定打开操作系统文件的参数  
                    RECORDS DELIMITED BY NEWLINE                <== 指定外部数据文件是一行为一个记录  
                    FIELDS TERMINATED by '|'                    <== 指定外部数据文件中以 '|' 为列分隔符  
                    MISSING FIELD VALUES ARE NULL                <== 指定外部数据文件中如有缺失则填写 NULL  
                    ( id, count, flag, comm,                    <== 指定外部表的每一列对应操作系统文件中的哪一列  
                        order_date CHAR DATE_FORMAT DATE MASK "mm/dd/yyyy"        <== 指定以何种方式读取操作操作系统文件中表示 date 的列  
                    )  
                )  
                LOCATION('test.csv')                        <== 指定操作系统文件名  
            )  
            REJECT LIMIT UNLIMITED                            <== 不加这一行可能会报错，所以一般都加上  
            ;  
    2.ORACLE_DATAPUMP 装载数据语法  
        # CREATE TABLE <table_name> (<column_list>  
            ORGANIZATION EXTERNAL(  
                TYPE ORACLE_DATAPUMP  
                DEFAULT DIRECTORY <dir_name>  
                LOCATION('emp.dmp' [, ...])                    <== 多个 dmp 文件自动启用并行  
            )  
            ;  
    3.ORACLE_DATAPUMP 卸载数据语法  
        # CREATE TABLE <table_name> [(<column_list>)]  
            ORGANIZATION EXTERNAL(  
                TYPE ORACLE_DATAPUMP                        <== 卸载数据只能使用 ORACLE_DATAPUMP 接口  
                DEFAULT DIRECTORY <dir_name>  
                ACCESS PARAMETERS (  
                    [COMPRESSION ENABLED BASIC]                <== 启用压缩，压缩功能需要付费  
                )  
                LOCATION('emp.dmp' [, ...])  
            )  
            [PARALLEL <N>]                                    <== 并行数要与 LOCATION 子句中的文件数相同  
            AS <sub_query>  
            ;
