7.1.1 SHELL

知识点–概念：SHELL

    1.SHELL 的意义
        # Kernel 在内存中是受保护的，不应该让用户直接控制 Kernel
        # 通过 Shell 将用户输入的指令与 Kernel 沟通，继而让 Kernel 控制硬件来正确的工作
    2.SHELL 的发展历史
        # 第一个流行的 shell，Bourne shell，简称 sh，纪念其创作者 Steven Bourne
        # 另一个流行的 shell，cshell，简称为 csh 
            - 由伯克利大学的 Bill Joy 设计依附于 BSD 版的 Unix 系统中的 shell
            - 这个 shell 的语法有点类似 C 语言，所以才得名为 C shell
            - 由于在学术界 Sun 主机势力相当的庞大，而 Sun 主要是 BSD 的分支之一，所以 C shell 也是另一个很重要的 shell
        # BASH，Bourne Again Shell
        # TCSH，CSHELL 的增强版
    2./etc/shells
        # 记录 Linux 支持的所有 shell
        # 用户能使用的 shell 必须能在 /etc/shells 中查询到
        # 某些服务在运作过程中，会检查用户的 shell 是否支持，就是查询 /etc/shells
        # 用户使用的 shell，记录在 /etc/passwd 中
    3./sbin/nologin
        # linux 默认的非法 shell
        # 比如 FTP 主机拒绝用户使用除 FTP 以外的资源，就可以给客户分配一些奇怪的 shell
        # 这些奇怪的 shell 需要写入到 /etc/shells 中
    4.login_shell 和 nologin_shell
        # login_shell
            - 进行完整的登录流程
            - 获得完整的环境设定
            - 例如，正常的从 tty1 登录或远程 ssh 登录
        # nologin_shell
            - 进行不完整的登录流程
            - 部分环境设定通过继承获得
            - 例如从 X window 中启动 terminal
            - 例如子 shell

        # seq 列表需要放在 $( ) 或 ` ` 中

            do
                <operations> [;]
            done [;]

            done [;]

        # <end>

            - 格式，例如 i<=100 或 mod(i,100)=15

            - 指定故常
            - 格式，例如 i=i+1 或 i++、i--

        # cat pay.txt | \

13.3.4 特殊权限

知识点–概念：文件和目录的特殊权限

    1.特殊权限有三个
        # SUID
        # SGID
        # SBIT
    2.文件的特殊权限设定后，若正常可使用，应该是小写字母
        # 如果出现大写字母，则代表权限条件不足
        # 例如设置了 SUID，但执行者没有对程序的 x 权限，此时 SUID 就会是大写的 S
        # 这种情况下，虽然设置了特殊权限，但特殊权限其实不起作用

知识点–概念：SUID

    1.Set UID
        # 仅对二进制程序文件有效，对 shell script 没用
        # 在二进制程序文件的拥有者的 x 权限出现 s 标志，称为 Set UID
        # 作用
            - 非程序的拥有者执行该程序时，能获得该程序拥有者的权限
            - 程序的运行时间结束后，执行者的权限又恢复为原来的状态
            - 执行者要拥有程序的 x 权限
    2.范例
        # 普通用户利用 passwd 来修改自己的密码
            - 正常情况下，普通用户无权修改 /etc/passwd、/etc/shadow 这几个文件
            - 但 passwd 设置了 SUID
            - 那么普通用户使用 passwd 指令时，就可以修改这几个文件了
            - 当 passwd 指令执行结束，普通用户的权限又恢复到原来的状态了

知识点–概念：SGID

    1.Set GID
        # 可以针对文件或目录来设定
        # 在文件组拥有者的 x 权限出现 s 标志，称为 Set GID
        # 程序文件 set UID 作用
            - 执行者在执行的过程中将会获得该程序组群的支持
            - 执行者要拥有该程序的 x 权限
        # 目录 set UID 作用
            - 用户若对于此目录具有 r 与 x 的权限时，该用户能够进入此目录
            - 用户在此目录下的有效组群 effective group将会变成该目录的组群
            - 当用户离开此目录时，其有效组群又恢复正常
    2.用途
        # 主要用在软件开发，让开发人员不用反复切换自己的主要组
        # 普通情况
            - 用户在某目录中建立的文件，必须切换适当的组群身份
            - 否则其他相关人员没法查看和修改，因为组群不同，要么就是所有人都能看和修改
        # set gid
            - 则用户在此文件下新建的文件，会自动的设置为目录的组拥有者

        #lvcreate -L N -T VGname/thinpoolname                   ，建立 thin pool
        #lvcreate -V N -T VGname/thinpoolname -n thinVolumename ，从 thin pool 中建立 thin Volume
        #lvcreate -s -L N -n snapshotname /dev/VGname/LVname    ，对指定的 LV 建立快照
    2.选项
        #-L，后面接容量，容量的单位可以是 M,G,T 等
            由于容量的最小单位为 PE，因此这个数量必须要是 PE 的倍数，若不相符，系统会自行计算最相近的容量
        #-l，后面可以接 PE 的个数而不是数量，需要人脑计算 PE 数
        #-n，后面接的就是 LV 的名字，名字自己指定
        #-T，建立 thin pool，必须同时指明 VG 的名字和 thin pool 的名字
        #-V，建立 thin Volume 时指定其宣告容量
        #-s，建立快照

知识点–概念：快照 snapshot

    1.快照与普通备份的区别，快照只备份修改过的数据
        #当有资料变动了，则将原始资料搬移到快照区，而修改写入到数据区
        #快照与数据区共享没有修改过的数据区
    2.快照与被快照的 LV 必须放在一个 VG 中
    3.快照与被快照的 LV 在文件系统层面的参数完全相同，包括 UUID
        #所以如果要挂载 xfs 文件系统的快照，要加上 nouuid 的选项，因为 xfs 不允许挂载相同 UUID 的文件系统
    4.鸟哥利用快照恢复的方法，个人不太赞同
        #先将快照利用 xfsdump 做成一个备份
        #再将 LV 和快照都卸载，并删除快照
        #对 LV 格式化，之后挂载
        #利用之前 xfsdump 做的备份，使用 xfsrestore 来恢复
    5.快照的另一个比较好用的功能，测试机
        #将原始 LV 当做备份，将快照当做测试机
        #所有操作和修改，都在快照里进行
        #如果测试机有问题，删除快照即可
        #如果需要新建一个测试机，就再新建一个快照即可
